# 新工程架构

## 目标

本工程以 Swift + UIKit 为主，采用 Clean Architecture Lite + MVVM。核心目标是：

- 目录、类型、页面命名与旧工程拉开距离。
- 后端路径、字段、错误码、鉴权 Header 保持兼容。
- 账号、会员、钻石、IAP、历史作品等用户资产模块优先隔离和测试。
- Swift Package 按需接入，不在骨架期引入无真实使用场景的依赖。

## 分层

| 层级 | 目录 | 责任 |
| --- | --- | --- |
| Application | `iPxavno/Application` | 启动、依赖装配、根路由 |
| Domain | `iPxavno/Domain` | 业务模型、Repository 协议、Service 协议 |
| Infrastructure | `iPxavno/Infrastructure` | 网络、Keychain、远端 Repository、埋点实现 |
| Features | `iPxavno/Features` | UIKit 页面和 ViewModel |
| Core | `iPxavno/Core` | 通用 UI、存储、错误、轻量工具 |

## 当前 Swift Package 策略

当前骨架没有接入第三方 Swift Package。原因：

- 网络层使用 `URLSession` 已能覆盖基础 JSON、上传签名 URL、SSE 可继续扩展。
- UIKit 页面先用 Auto Layout 和系统 SF Symbols，避免无必要依赖。
- 账号凭证先用系统 Keychain，不需要第三方安全存储包。

后续建议按真实需求接入：

| 场景 | 候选包 | 接入时机 |
| --- | --- | --- |
| 图片缓存与渐进加载 | Kingfisher / Nuke | 远端图片列表和历史作品大规模上线时 |
| 响应式状态流 | CombineExt 或保持原生 Combine | ViewModel 状态复杂到跨模块共享时 |
| 数据库 | GRDB | 历史作品、任务缓存、原图关联需要稳定迁移时 |
| 布局 DSL | SnapKit | 页面数量上来且团队统一 DSL 时 |
| StoreKit 封装 | RevenueCat 或自研 StoreKit 2 | 如果业务允许第三方订阅服务；否则保持自研 |

## 命名策略

旧工程概念不直接搬运到类型名：

- 首页模块命名为 `Discover`。
- AI 制作模块命名为 `Creation`。
- 用户模块命名为 `Profile`。
- 网络入口命名为 `APIClient`。
- 账号凭证入口命名为 `SessionVault`。

后端字段通过 `CodingKeys` 兼容，例如 `filter_id` 映射到 `CreativeTemplate.id`，`category_id` 映射到 `CreativeKind`。
