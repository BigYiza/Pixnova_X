# 制作参数配置与钻石计算

本文档梳理制作页高级参数配置相关逻辑，重点关注 `DiamondsExpenseHandler` 如何接入 UI、计算钻石消耗、把结果反馈给 UI，以及参数如何保存并传给后续制作接口。

## 数据模型与来源

制作卡片的基础价格和高级参数都挂在 `FilterItem` 上：

- `diamonds`：基础钻石消耗，兜底值为 `10`。
- `diamondsExpenseParam`：由后端 `custom_parameter` 映射而来的 `DiamondsExpenseModel`。

`DiamondsExpenseModel` 包含两部分：

- `parameters` -> `params`：参数分组列表，用于 UI 展示和价格计算。
- `restricts` -> `restricts`：参数联动约束，用于在某个选项变化后自动修正其他不兼容选项。

每个 `ExpenseParamItem` 表示一个参数组：

- `key`：最终传给后端时的参数名。
- `title`：弹窗标题，也是自动切参 toast 中的参数名。
- `icon`：参数入口上的图标。
- `selected` -> `defaultSelected`：后端指定的默认选中值。
- `values`：可选项列表。
- `selectedIndex`：客户端当前选中的下标，默认 `0`，后续由默认值或用户选择更新。

每个 `ExpenseParamItemValue` 表示一个可选项：

- `label`：UI 展示文案。
- `value`：最终传给后端的真实值。
- `times`：该选项的价格倍率。
- `calculate_method` -> `calculateMehodId`：特殊计费方式标识。
- `duration_times` -> `durationTimes`：视频增强类计费会用到的视频时长倍率。

`DiamondsExpenseHandler` 内部还保留了一个 `/api/query_custom_parameter` 请求方法 `requeseDimondsExpenseInfo()`，可以按 `filter_id` 单独拉高级参数。但当前主流程使用的是 `FilterItem.diamondsExpenseParam`，这个私有方法没有被调用。

## Handler 初始化

制作页初始化或当前模板变化时，会配置 `DiamondsExpenseHandler`：

```swift
handler.filterItem = filterItem
handler.expense = filterItem.diamonds ?? 10
handler.setCurrentDiamondsExpenseMondel(model: filterItem.diamondsExpenseParam)
```

`setCurrentDiamondsExpenseMondel` 做三件事：

1. 如果没有配置，`expenseModel = nil`，后续价格回退到基础价。
2. 如果有配置，通过 `model.toJSON()` 重新构造一份 `DiamondsExpenseModel` 副本。
3. 调用 `configAppUICodeDefault()` 按后端 `selected` 设置默认 `selectedIndex`，再调用 `refreshPriceInCurrentParam()` 计算当前价格。

这里的“副本”很重要：UI 修改的是 handler 内部的 `expenseModel.params[].selectedIndex`，不会直接改掉 `FilterItem.diamondsExpenseParam` 原始配置。因此同一个模板重新进入或 handler 重建时，会重新回到后端默认选项。

## UI 如何触发计算

### 老 AI Video 制作页

`VideoCreatePage` 在 `init(filterItem:)` 中创建并初始化 `@State private var diamondsExpensenHandler`，然后通过 binding 传给 `VideoCreateContentView`。

页面里如果 `diamondsExpensenHandler.expenseModel?.params` 不为空，就展示 `VideoConditionListView`。用户点击某个参数入口时：

1. `VideoConditionListView` 把当前参数组和入口 frame 回传。
2. 页面设置 `currentPopParam`、`conditionViewFrame`，打开 `VideoConditionPopupView`。
3. 用户在弹窗中点击某个 value。
4. `VideoConditionPopupView` 先把 `param.selectedIndex = index`，再通过 `onItemTapped(index)` 回传。
5. 页面取到 `targetItem = currentPopParam?.values?[index]`，调用：

```swift
diamondsExpensenHandler.refreshDiamondExpenseWith(targetItem: targetItem)
```

### 自定义视频制作页

`CustomVideoEditPromptViewModel` 持有 `var diamondsExpensenHandler = DiamondsExpenseHandler()`。

这些场景会刷新 handler 配置：

- 切换 `promptType`，例如 Image To Video / Text To Video。
- 切换 `imageCountType`，例如单图 / 多图。
- 设置 `usingFilter`，例如用户选择推荐模板或从外部带入模板。
- 初始化入口指定了 `preferredFilterItem` 时。

刷新逻辑是 `refreshDiamondsExpensenHandlerConfig()`：

```swift
if currentFilter.id != handler.filterItem?.id {
    handler.filterItem = usingFilter
    handler.expense = usingFilter.diamonds ?? 10
    handler.setCurrentDiamondsExpenseMondel(model: currentFilter.diamondsExpenseParam)
}
```

UI 侧和老 AI Video 一样，用 `VideoConditionListView` 展示，用 `VideoConditionPopupView` 选择，选择后调用 `viewModel.diamondsExpensenHandler.refreshDiamondExpenseWith(targetItem:)`。

### 视频增强延伸场景

`VideoResolutionViewModel` 也复用了 `DiamondsExpenseHandler`。它在初始化时额外写入：

```swift
expenseHandler.optionalParam_Duration = videoDuration
```

当某个选项的 `calculate_method == "current and ugc duration multiplier"` 时，价格计算会额外乘以 `duration_times` 和 `optionalParam_Duration`。

## Handler 计算逻辑

用户选择某个参数值后，入口是：

```swift
refreshDiamondExpenseWith(targetItem:)
```

内部执行顺序：

1. `relationshipWithParam(targetItem:)`
2. `relationshipCheckInMinorParam(mainTargetItem:)`
3. `refreshPriceInCurrentParam()`

### 1. 当前选项与直接联动

`relationshipWithParam` 先看是否有 `restricts`：

- 没有 `restricts`：在所有 `params` 里找到包含 `targetItem` 的参数组，把该组 `selectedIndex` 更新为目标下标。
- 有 `restricts`：找到 `restrict.value == targetItem.value` 的约束，然后检查约束中列出的其他参数当前值是否在允许范围内。如果不在，自动切到允许值列表的第一个值，并触发 toast。

自动切参 toast 文案格式：

```text
[{param.title}] was automatically updated to [{selected.label}] based on model requirements.
```

### 2. 反向检查次级参数

`relationshipCheckInMinorParam` 用来处理“用户选了主参数，但已有的次级参数可能反过来限制主参数”的情况。

逻辑大致是：

1. 找到包含 `mainTargetItem` 的主参数组。
2. 其他参数组都视为 minor 参数。
3. 对每个 minor 参数当前选中值，查它自己的 `restricts`。
4. 如果 minor 当前值限制了主参数，而主参数的新值不在允许范围内，就在 minor 参数可选项里找一个能兼容主参数新值的项，并自动切过去。
5. 自动切换后同样触发 toast。

这一步保证不仅“主参数约束次参数”生效，“次参数已选值反向约束主参数”的场景也能被修正。

### 3. 价格计算

`refreshPriceInCurrentParam` 的核心公式是：

```text
expense = Int((filterItem.diamonds ?? 10) * 所有已选参数倍率乘积)
```

每个参数组取当前 `selectedIndex` 对应的 `ExpenseParamItemValue`：

- 普通选项：乘 `Double(times ?? "1.0") ?? 1.0`。
- 视频增强特殊选项：如果 `calculateMehodId == ExpenseParamItemValue.VideoBoostId`，先额外乘：

```text
(Double(durationTimes ?? "1.0") ?? 1.0) * (optionalParam_Duration ?? 1.0)
```

然后仍会继续乘 `times`。

如果没有 `expenseModel`，价格回退为 `filterItem?.diamonds ?? 10`。

## 计算后如何反馈到 UI

`DiamondsExpenseHandler` 是 `@Observable`，SwiftUI 页面直接读取它的状态，不需要显式回调。

主要反馈点：

- 制作按钮右侧钻石数读取 `handler.expense`，重新计算后 UI 自动刷新。
- 参数入口 `VideoConditionView` 读取 `param.values?[param.selectedIndex].label`，`selectedIndex` 变化后入口文案自动刷新。
- 参数弹窗 `VideoConditionPopupView` 用 `param.selectedIndex == index` 控制选中态。
- 自动联动切参时，handler 写入 `toastContent` 和 `toastNeeded`，页面通过 `.toast(str:presentingToast:)` 展示提示。

生成按钮点击时会用最新的 `handler.expense` 做余额校验：

```swift
if AccountManager.default.currentAccount.diamonds < handler.expense {
    isPresentVideoTimePage = true
} else {
    // 进入上传/生成
}
```

余额不足时打开钻石购买弹窗；余额足够时继续组装 `UploadModel`。

## 参数如何保存

当前选中参数只保存在运行时对象里，主要有两层：

1. `DiamondsExpenseHandler.expenseModel.params[].selectedIndex`
   - 保存当前页面内每个参数组的选中下标。
   - 会被用户选择、默认值配置、联动修正共同更新。
   - 因为 `expenseModel` 是从 `FilterItem.diamondsExpenseParam` clone 出来的，所以这是页面级临时状态。

2. `UploadModel.externalArgs`
   - 用户点击生成且余额足够时，页面把当前 `selectedIndex` 转成 `[key: value]` 字典。
   - 这个字典绑定到本次制作任务。

组装逻辑在多个页面里一致：

```swift
var params: [String: Any] = [:]
handler.expenseModel?.params?.forEach {
    if let values = $0.values,
       $0.selectedIndex < values.count,
       let value = values[$0.selectedIndex].value,
       let key = $0.key {
        params[key] = value
    }
}
uploadModel.externalArgs = params
```

客户端没有把这些高级参数选择写入 UserDefaults、数据库或 `FilterItem` 原对象。页面销毁、切到另一个 filter 或 handler 重新初始化后，会重新按后端默认值配置。

## 后续如何使用

生成流程中，`UploadModel` 进入 `UploadTask`，再进入 `UploadManager.generaImage/generaVideo`。

创建任务时，`UploadManager.createTask(urls:model:)` 会把参数组装为 `/api/image_generation` 的请求体：

```swift
var params: [String: Any] = [
    "urls": urls,
    "filter_id": model.filterId
]

if let prompt = model.prompt, prompt.count > 0 {
    params["prompt"] = prompt
}

if let negativePrompt = model.negativePrompt {
    params["negative_prompt"] = negativePrompt
}

if let externalArgs = model.externalArgs {
    params["external_args"] = externalArgs
}

if let combineConfigs = model.combineConfigs {
    params["combine_configs"] = combineConfigs
}
```

最终传给后端的高级参数形态是：

```json
{
  "external_args": {
    "{ExpenseParamItem.key}": "{ExpenseParamItemValue.value}"
  }
}
```

也就是说，同一份参数选择承担两类职责：

- 客户端侧：计算预估钻石消耗、展示按钮价格、余额不足拦截、参数兼容性自动修正。
- 服务端侧：作为 `external_args` 控制本次生成任务实际使用的模型参数。

## 主流程图

```mermaid
flowchart TD
    A[FilterItem] --> B[diamonds 基础价]
    A --> C[diamondsExpenseParam/custom_parameter]
    C --> D[DiamondsExpenseHandler clone expenseModel]
    D --> E[configAppUICodeDefault 设置 selectedIndex]
    E --> F[refreshPriceInCurrentParam 计算 expense]
    F --> G[UI 展示参数入口和 Generate 钻石数]
    G --> H[用户打开 VideoConditionPopupView 并选择参数值]
    H --> I[refreshDiamondExpenseWith]
    I --> J[参数联动与自动切参]
    J --> K[重新计算 expense]
    K --> G
    G --> L[点击 Generate]
    L --> M{余额是否足够}
    M -- 否 --> N[打开 DiamondPurchaseView]
    M -- 是 --> O[按 selectedIndex 组装 UploadModel.externalArgs]
    O --> P[UploadManager.createTask]
    P --> Q[/api/image_generation external_args]
```

## 注意点

- `ExpenseParamItemValue` 的 `Hashable/Equatable` 只比较 `label` 和 `times`，不比较 `value`。当前主要用于 SwiftUI `ForEach(id: \.element)`，如果同一参数组出现相同 `label + times` 但不同 `value`，可能导致 UI diff 不稳定。
- `VideoConditionPopupView` 会先直接写 `param.selectedIndex = index`，再回调 handler。没有 `restricts` 的情况下 handler 会再次确认同一选中值；有 `restricts` 时，handler 主要负责修正其他参数。
- `configAppUICodeDefault()` 只在初始化配置时按 `selected` 找默认值。后续用户选择不会回写 `defaultSelected`。
- `refreshDiamondsExpensenHandlerConfig()` 只有在 filter id 变化时才重建 handler 配置。同一个 filter 下如果后端配置对象被热更新但 id 不变，当前页面不会重新加载。
- `requeseDimondsExpenseInfo()` 方法名存在拼写问题且当前未使用。如果后续要从详情接口动态刷新参数，需要显式调用并考虑成功后是否也要执行默认值配置和价格刷新。
